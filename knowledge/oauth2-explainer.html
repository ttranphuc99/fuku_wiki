OAuth

Authorization framework that lets third-party apps access protected resources — without ever touching credentials. Deep-dive: flows, tokens, Spring Boot integration.

01 // Core Concepts

The Four
Actors

👤
Resource Owner
The human user who owns the data and grants access permissions. They authenticate with the Authorization Server and decide what the Client can access.
e.g. End User / You
📱
Client
The application requesting access. Could be your Spring Boot app, SPA, or mobile app. It requests tokens and uses them to access resources on behalf of the owner.
e.g. Your Web App
🔐
Authorization Server
Issues access tokens after authenticating the Resource Owner and obtaining authorization. It's the trust anchor of the entire OAuth2 ecosystem.
e.g. Keycloak, Auth0, Okta
🗄️
Resource Server
Hosts the protected API endpoints. Validates access tokens on every request and enforces scope-based access control. Your REST API is typically here.
e.g. Your REST API
⚠️ Critical Distinction
OAuth2 ≠ Authentication. OAuth2 is an authorization protocol — it grants access, not identity. For authentication (login), you need OpenID Connect (OIDC), which is built on top of OAuth2 and adds an ID Token + /userinfo endpoint.
02 // Authorization Flows

Step-by-Step
Flows

Best for: Traditional web apps with a secure backend server (Spring Boot MVC, etc.)
1
User triggers login
User clicks "Login" — Client redirects browser to Authorization Server with required params.
GET /authorize? response_type=code &client_id=my-app &redirect_uri=https://app.com/callback &scope=openid profile email &state=random_csrf_token
2
User authenticates & consents
Auth Server presents login page. User enters credentials, grants requested scopes.
3
Auth Server returns code
Redirects browser back to redirect_uri with a short-lived authorization code.
GET /callback?code=SplxlOBeZQQYbYS6WxSbIA&state=random_csrf_token
4
Client exchanges code for tokens
Server-to-server POST — client_secret never exposed to browser. Returns access + refresh tokens.
POST /token grant_type=authorization_code &code=SplxlOBeZQQYbYS6WxSbIA &client_id=my-app &client_secret=super-secret &redirect_uri=https://app.com/callback
5
Access protected resources
Client includes access token in every API request as a Bearer token.
GET /api/orders Authorization: Bearer eyJhbGciOiJSUzI1NiJ9...
Best for: SPAs, Mobile apps — cannot safely store a client_secret. PKCE replaces it cryptographically.
1
Generate PKCE pair
Client generates a cryptographically random code_verifier and derives the code_challenge.
// In your SPA/mobile app: code_verifier = random(43..128 chars) code_challenge = BASE64URL(SHA256(code_verifier))
2
Send challenge with auth request
Include code_challenge and method in the initial redirect. The verifier stays secret client-side.
GET /authorize? response_type=code &client_id=spa-app &code_challenge=E9Melhoa2OwvFrEMTJguCHaoeK... &code_challenge_method=S256 &scope=openid profile
3
Exchange code with verifier (no secret!)
Auth Server verifies: SHA256(code_verifier) === code_challenge. No client secret needed.
POST /token grant_type=authorization_code &code=SplxlOBeZQQYbYS6WxSbIA &client_id=spa-app &code_verifier=dBjftJeZ4CVP-mB92K27uhbUJU1p... // ↑ No client_secret!
Best for: Microservice-to-microservice communication. No user involved — machine identity only.
1
Service requests token directly
No user redirection. Service authenticates itself with its own credentials.
POST /token grant_type=client_credentials &client_id=order-service &client_secret=svc-secret-xyz &scope=inventory:read payments:write
2
Auth Server returns access token
No refresh token issued — services just re-request tokens when they expire.
// Response: { "access_token": "eyJhbGci...", "token_type": "Bearer", "expires_in": 3600, "scope": "inventory:read payments:write" }
3
Call target microservice
Token carries service identity and scopes. Target service validates and checks scopes.
GET /api/inventory/123 Authorization: Bearer eyJhbGci... // Target validates token, checks SCOPE_inventory:read
Best for: Smart TVs, CLI tools, IoT devices — no browser/keyboard available.
1
Device requests a code
Device POSTs to the device authorization endpoint to get a user code and verification URL.
POST /device/code client_id=tv-app&scope=openid profile // Returns: { "device_code": "GmRhmhcxhwAzkoEqiMEg_DnyEysNkuNhszIySk9eS", "user_code": "WDJB-MJHT", "verification_uri": "https://auth.com/activate", "expires_in": 900, "interval": 5 }
2
User visits URL on another device
TV displays: "Visit auth.com/activate and enter code WDJB-MJHT". User does this on phone/laptop.
3
Device polls for token
Meanwhile, device polls the token endpoint every 5 seconds until user completes auth.
POST /token grant_type=urn:ietf:params:oauth:grant-type:device_code &device_code=GmRhmhcxhwAzkoEqiMEg_DnyEysNkuNhszIySk9eS &client_id=tv-app // Receives: authorization_pending → keep polling → success!

// Header — Algorithm & Type
{
  "alg": "RS256",   // RS256 = RSA + SHA256 (asymmetric, preferred in prod)
  "typ": "JWT"      // Token type
}
// Payload — Claims (not encrypted, base64url encoded)
{
  "sub": "user_123",              // Subject — unique user identifier
  "iss": "https://auth.example.com", // Issuer — who issued this token
  "aud": "my-api",                // Audience — intended recipient
  "exp": 1700000000,              // Expiry (Unix epoch) — ALWAYS validate!
  "iat": 1699996400,              // Issued At
  "roles": ["USER", "ADMIN"],    // Custom claim — your roles
  "scope": "read write"           // OAuth2 scopes granted
}
// Signature — Tamper Detection
RSASHA256(
  base64url(header) + "." + base64url(payload),
  // Signed with Auth Server's PRIVATE key
  // Verified with Auth Server's PUBLIC key (JWKS endpoint)
  privateKey
)

04 // Grant Types

Choose Your
Flow

🔐 Recommended
Authorization Code
The gold standard for web apps with a backend. The client_secret stays server-side, never touching the browser.
Use for: Spring Boot SSR Apps Node.js
🛡️ + PKCE Required
Auth Code + PKCE
Same as Auth Code but replaces client_secret with a cryptographic proof. Mandatory for any public client.
Use for: React SPA Angular Mobile
⚙️ Machine-to-Machine
Client Credentials
No user involved. A service authenticates with its own identity. Perfect for microservice communication in your Spring ecosystem.
Use for: Microservices Daemons CI/CD
📺 Limited Input Devices
Device Code
Designed for devices that can't open a browser. User completes auth on a secondary device while the primary polls for the token.
Use for: Smart TV CLI IoT
⛔ Deprecated
Password Grant
Client collects username/password directly. Completely defeats the purpose of OAuth2. Removed in OAuth 2.1. Never use.
AVOID AT ALL COSTS
🔄 Token Refresh
Refresh Token Grant
Not a standalone flow — used to silently obtain a new access token when the current one expires, without re-prompting the user.
Use for: Token renewal Long sessions
Flow Selector
Is a user (human) involved?
NO →
Client Credentials
Machine-to-machine. Service authenticates with its own client_id + client_secret.
YES → Does the app have a secure backend?
YES →
Auth Code Flow
NO (SPA/Mobile) →
Auth Code + PKCE
NO BROWSER →
Device Code Flow
CLI, Smart TV, IoT. User authorizes on a secondary device.

@Configuration
@EnableWebSecurity
@EnableMethodSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(AbstractHttpConfigurer::disable) // Stateless API
            .sessionManagement(s -> s
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/actuator/health", "/public/**").permitAll()
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .requestMatchers("/api/orders").hasAuthority("SCOPE_read:orders")
                .anyRequest().authenticated()
            )
            .oauth2ResourceServer(oauth2 -> oauth2
                .jwt(jwt -> jwt
                    .jwtAuthenticationConverter(jwtAuthConverter())
                )
            );
        return http.build();
    }

    @Bean
    public JwtAuthenticationConverter jwtAuthConverter() {
        var rolesConverter = new JwtGrantedAuthoritiesConverter();
        rolesConverter.setAuthoritiesClaimName("roles"); // custom claim name
        rolesConverter.setAuthorityPrefix("ROLE_");

        var converter = new JwtAuthenticationConverter();
        converter.setJwtGrantedAuthoritiesConverter(rolesConverter);
        return converter;
    }
}

@RestController
@RequestMapping("/api/orders")
public class OrderController {

    // Method-level security via @PreAuthorize
    @GetMapping
    @PreAuthorize("hasAuthority('SCOPE_read:orders')")
    public List<Order> getOrders(@AuthenticationPrincipal Jwt jwt) {
        var userId = jwt.getSubject();      // "sub" claim
        var email = jwt.getClaimAsString("email");
        var roles = jwt.getClaimAsStringList("roles");

        return orderService.findByUserId(userId);
    }

    @PostMapping
    @PreAuthorize("hasRole('USER') and hasAuthority('SCOPE_write:orders')")
    public Order createOrder(
            @RequestBody CreateOrderRequest req,
            @AuthenticationPrincipal Jwt jwt) {
        return orderService.create(req, jwt.getSubject());
    }

    // Admin only
    @DeleteMapping("/{id}")
    @PreAuthorize("hasRole('ADMIN')")
    public void deleteOrder(@PathVariable Long id) {
        orderService.delete(id);
    }
}
/**
 * Token-propagating WebClient for microservice-to-microservice calls.
 * Uses Client Credentials flow — order-service calls inventory-service.
 */
@Configuration
public class WebClientConfig {

    @Bean
    public WebClient inventoryClient(
            OAuth2AuthorizedClientManager authorizedClientManager) {

        var interceptor = new ServletOAuth2AuthorizedClientExchangeFilterFunction(
                authorizedClientManager);

        // Use 'order-to-inventory' registration from application.yml
        interceptor.setDefaultClientRegistrationId("order-to-inventory");

        return WebClient.builder()
                .baseUrl("https://inventory-service.internal")
                .apply(interceptor.oauth2Configuration())
                .build();
    }
}

// Usage in OrderService:
@Service
public class OrderService {

    private final WebClient inventoryClient;

    public StockLevel checkStock(String productId) {
        return inventoryClient.get()
                .uri("/stock/{id}", productId)
                .retrieve()
                .bodyToMono(StockLevel.class)
                .block();
        // Token auto-fetched + cached by Spring Security!
    }
}
06 // Security Hardening

What To Do
& What Not To

✅
Short-lived access tokens

Keep access tokens to 5–60 minutes. Limits blast radius if stolen. Use refresh tokens for seamless renewal.

❌
Never store tokens in localStorage

XSS attacks can steal localStorage. Use HttpOnly + Secure + SameSite=Strict cookies for token storage.

✅
Always use HTTPS

Tokens are Bearer credentials. A token over HTTP = your credentials in plaintext. Non-negotiable.

❌
Never expose client_secret in frontend

SPAs and mobile apps cannot keep secrets. Use PKCE instead — it provides the same protection cryptographically.

✅
Validate all JWT claims

Always verify: iss (issuer), aud (audience), exp (expiry), nbf (not before). Spring does this automatically.

❌
Don't use Password Grant

It forces users to give their credentials to your app, bypassing the Auth Server. Removed in OAuth 2.1 for good reason.

✅
Rotate refresh tokens

Issue a new refresh token on every use (Refresh Token Rotation). Enables detection of token theft via reuse detection.

✅
Use state parameter (CSRF protection)

Generate a random state param on redirect. Verify it on callback. Prevents CSRF attacks on your OAuth flow.

OAuth2 vs OpenID Connect (OIDC)

OAuth2 OpenID Connect (OIDC)
Purpose Authorization (access control) Authentication (identity)
Tokens Access Token, Refresh Token + ID Token (contains user info)
User Info No standard /userinfo endpoint
Required Scope Custom scopes Must include openid
Answers "What can this app do?" "Who is this user?"
