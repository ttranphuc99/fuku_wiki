01
Authentication Flow

How OIDC Works

OIDC extends the OAuth2 Authorization Code flow with one critical addition: alongside the access token, the Authorization Server issues an ID Token — a signed JWT containing the user's verified identity.

1
Client initiates authentication
User clicks "Sign In". Client redirects to the Authorization Server's /authorize endpoint. The critical difference from plain OAuth2: scope must include openid.
GET /authorize? response_type=code &client_id=my-app &redirect_uri=https://app.com/callback &scope=openid profile email ← openid is mandatory! &state=csrf_token_xyz &nonce=random_replay_prevention_value
2
User authenticates at Authorization Server
Auth Server presents a login form. User enters credentials. Auth Server verifies them, establishes a session, and records consent. The nonce from step 1 will be embedded in the ID Token.
3
Auth Server redirects with authorization code
Browser is redirected back to your redirect_uri with a short-lived authorization code. State parameter must be validated to prevent CSRF.
GET /callback? code=SplxlOBeZQQYbYS6WxSbIA &state=csrf_token_xyz ← validate this matches!
4
Client exchanges code for tokens (key difference!)
Server-to-server token exchange. Auth Server returns three things: an access token, a refresh token, AND — unique to OIDC — an ID Token.
POST /token grant_type=authorization_code &code=SplxlOBeZQQYbYS6WxSbIA &client_id=my-app &client_secret=super-secret &redirect_uri=https://app.com/callback // Response: { "access_token": "eyJhbGci...", ← for API access "refresh_token": "dGhpcyBp...", ← for renewal "id_token": "eyJhbGci...", ← OIDC identity JWT ✨ "token_type": "Bearer", "expires_in": 3600 }
5
Client validates the ID Token
This step is mandatory and critical. Client must validate the ID Token before trusting any identity claims. Failure to validate = security vulnerability.
// Must validate: ✓ Signature — using Auth Server's public key (JWKS) ✓ iss claim — must match expected issuer ✓ aud claim — must match your client_id ✓ exp claim — must not be expired ✓ nonce claim — must match what was sent in step 1 ✓ iat claim — must be recent (prevent old token replay)
6
Optionally fetch UserInfo
For additional claims not in the ID Token, call the /userinfo endpoint using the access token. Returns a JSON object with the requested profile data.
GET /userinfo Authorization: Bearer <access_token> // Response: { "sub": "user_1a2b3c4d", "name": "Jane Doe", "email": "jane@corp.com", "picture": "https://cdn.example.com/avatar.jpg", "given_name": "Jane", "family_name": "Doe", "locale": "en-US" }
7
Session established — user is authenticated
Client establishes a local session based on the validated identity. Use the access token for API calls. Store refresh token securely for session renewal.

Catalogue

OIDC defines a standard set of claims for user identity. Hover any card to learn its purpose. These are returned in the ID Token and/or from the /userinfo endpoint.

04
Access Granularity

OIDC Scopes

Scopes determine which claims are available in the ID Token and from the UserInfo endpoint. Request only what you need — principle of least privilege.

openid
Required. Signals this is an OIDC request (not plain OAuth2). Without this scope, no ID Token is issued and no /userinfo endpoint is available. Always include this.
sub iss aud exp iat
profile
Access to the user's basic profile information. Returns name components, picture URL, preferred username, website, gender, birthdate, timezone, locale, and last updated time.
name given_name family_name middle_name nickname picture preferred_username locale zoneinfo updated_at
email
Access to the user's email address and its verification status. Always check email_verified — an unverified email could belong to anyone who claimed it during registration.
email email_verified
phone
Access to the user's phone number and its verification status. Similarly, check phone_number_verified before using for authentication purposes.
phone_number phone_number_verified
address
Access to the user's physical mailing address. Returns a JSON structure with sub-fields. Typically needed only for e-commerce, delivery, or compliance flows.
address.formatted address.street_address address.locality address.region address.postal_code address.country
offline_access
Requests a refresh token to enable long-lived sessions without re-prompting the user. Auth Servers may require explicit user consent for this scope. Required for background/daemon processes acting on behalf of a user.
refresh_token issued

Self-Describing Servers

Discovery
Document

Every OIDC provider publishes a JSON metadata document at a well-known URL. This single document tells clients everything they need — endpoints, supported algorithms, JWKS location. Spring Security fetches this automatically at startup.

🔍 Well-Known Endpoint
{issuer-uri}/.well-known/openid-configuration
Fetch this URL to bootstrap your OIDC client. Spring Security uses the issuer-uri property to auto-discover everything else.
Popular providers
Keycloak: /realms/{realm}/.well-known/openid-configuration
Google: accounts.google.com/.well-known/openid-configuration
Auth0: {domain}/.well-known/openid-configuration
Okta: {domain}/oauth2/default/.well-known/openid-configuration

@Configuration
@EnableWebSecurity
@EnableMethodSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/public/**", "/error").permitAll()
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()  // All others require login
            )

            // OIDC login — Spring handles the entire flow
            .oauth2Login(oidc -> oidc
                .loginPage("/login")              // Custom login page URL
                .defaultSuccessUrl("/dashboard", true)
                .userInfoEndpoint(userInfo -> userInfo
                    .oidcUserService(oidcUserService())  // Custom user loading
                )
            )

            // OIDC logout — invalidates session + tells IdP
            .logout(logout -> logout
                .logoutSuccessHandler(new OidcClientInitiatedLogoutSuccessHandler(
                    clientRegistrationRepository))
                .invalidateHttpSession(true)
                .clearAuthentication(true)
                .deleteCookies("JSESSIONID")
            );

        return http.build();
    }

    @Bean
    public OidcUserService oidcUserService() {
        OidcUserService delegate = new OidcUserService();

        return oidcUserRequest -> {
            // Let the default service load the user from /userinfo
            OidcUser oidcUser = delegate.loadUser(oidcUserRequest);

            // Extract roles from ID Token custom claim
            List<String> roles = oidcUser.getIdToken()
                .getClaimAsStringList("roles");

            var authorities = roles.stream()
                .map(r -> new SimpleGrantedAuthority("ROLE_" + r))
                .collect(Collectors.toSet());

            // Merge default authorities + custom roles
            authorities.addAll(oidcUser.getAuthorities());

            return new DefaultOidcUser(authorities,
                oidcUser.getIdToken(), oidcUser.getUserInfo());
        };
    }
}
@RestController
public class UserController {

    /**
     * Get the currently authenticated user's identity from the OIDC session.
     * @AuthenticationPrincipal injects the OidcUser loaded after login.
     */
    @GetMapping("/me")
    public UserProfileResponse getMe(
            @AuthenticationPrincipal OidcUser oidcUser) {

        // Access ID Token claims directly
        String sub    = oidcUser.getSubject();        // Stable user ID
        String email  = oidcUser.getEmail();          // From email scope
        String name   = oidcUser.getFullName();       // From profile scope
        String pic    = oidcUser.getPicture();        // Profile picture URL
        Boolean ver   = oidcUser.getEmailVerified(); // Always check this!
        Instant auth  = oidcUser.getAuthenticatedAt(); // auth_time claim

        // Access raw claims for custom fields (e.g., from Keycloak)
        List<String> roles = oidcUser.getIdToken()
                .getClaimAsStringList("roles");

        // Access UserInfo attributes (fetched from /userinfo endpoint)
        String locale = oidcUser.getUserInfo()
                .getClaimAsString("locale");

        return new UserProfileResponse(sub, email, name, roles);
    }

    // Step-up authentication: require fresh authentication for sensitive ops
    @PostMapping("/account/delete")
    @PreAuthorize("hasRole('USER')")
    public ResponseEntity<?> deleteAccount(
            @AuthenticationPrincipal OidcUser oidcUser) {

        Instant authTime = oidcUser.getAuthenticatedAt();
        boolean recentAuth = authTime.isAfter(Instant.now().minusSeconds(300));

        if (!recentAuth) {
            // Force re-authentication for sensitive operation
            return ResponseEntity.status(401)
                .header("WWW-Authenticate", "Bearer realm=\"step-up\"")
                .build();
        }

        userService.deleteUser(oidcUser.getSubject());
        return ResponseEntity.noContent().build();
    }
}
/**
 * OIDC supports two logout flows:
 *
 * 1. RP-Initiated Logout: Your app redirects user to the IdP logout endpoint.
 *    IdP clears its session and redirects back to your post_logout_redirect_uri.
 *
 * 2. Back-Channel Logout: IdP proactively notifies your app when a session
 *    ends elsewhere (e.g., user logs out from another app in the same SSO session).
 */
@Configuration
public class OidcLogoutConfig {

    @Bean
    public SecurityFilterChain logoutChain(HttpSecurity http,
            ClientRegistrationRepository repo) throws Exception {

        // RP-Initiated Logout (standard)
        OidcClientInitiatedLogoutSuccessHandler logoutHandler =
            new OidcClientInitiatedLogoutSuccessHandler(repo);

        // Where to land after IdP logout completes
        logoutHandler.setPostLogoutRedirectUri("https://app.corp.com/logged-out");

        http.logout(logout -> logout
            .logoutUrl("/logout")
            .logoutSuccessHandler(logoutHandler)
            .invalidateHttpSession(true)
            .clearAuthentication(true)
            .deleteCookies("JSESSIONID", "remember-me")
        );

        return http.build();
    }
}

// Back-Channel Logout (Spring Security 6.2+)
// application.yml:
// spring.security.oauth2.client.provider.keycloak.back-channel-logout-uri:
//   {baseUrl}/logout/connect/back-channel/{registrationId}
@Component
public class BackChannelLogoutHandler {

    // Spring Security 6.2+ handles this automatically when configured.
    // IdP POSTs a logout_token (JWT) to your back-channel URI.
    // Spring validates it and invalidates the corresponding session.

    // For older setups, you can implement SessionRegistry to track sessions by sub:
    @Autowired
    private SessionRegistry sessionRegistry;

    public void handleLogout(String sub) {
        sessionRegistry.getAllPrincipals().stream()
            .filter(p -> ((OidcUser) p).getSubject().equals(sub))
            .flatMap(p -> sessionRegistry.getAllSessions(p, false).stream())
            .forEach(SessionInformation::expireNow);
    }
}
/**
 * Best practice: auto-provision users in your DB on first OIDC login.
 * Use 'sub' as the stable external ID, never email.
 */
@Service
@Transactional
public class UserProvisioningService {

    @Autowired private UserRepository userRepo;
    @Autowired private AuditLogService auditLog;

    /**
     * Called after successful OIDC login.
     * Upserts the user based on their 'sub' claim.
     */
    public AppUser provisionFromOidc(OidcUser oidcUser) {
        String sub   = oidcUser.getSubject();  // Stable ID — use as FK
        String email = oidcUser.getEmail();

        // ⚠️  email_verified check before using email
        if (!Boolean.TRUE.equals(oidcUser.getEmailVerified())) {
            throw new UnverifiedEmailException("Email not verified for sub: " + sub);
        }

        return userRepo.findByOidcSub(sub)
            .map(existing -> {
                // Update profile if changed at IdP
                existing.setEmail(email);
                existing.setDisplayName(oidcUser.getFullName());
                existing.setLastLoginAt(Instant.now());
                return userRepo.save(existing);
            })
            .orElseGet(() -> {
                // First-time login — provision new user
                AppUser newUser = AppUser.builder()
                    .oidcSub(sub)                      // external ID from IdP
                    .email(email)
                    .displayName(oidcUser.getFullName())
                    .picture(oidcUser.getPicture())
                    .roles(Set.of("USER"))
                    .createdAt(Instant.now())
                    .build();

                auditLog.log("USER_PROVISIONED", sub, "First OIDC login");
                return userRepo.save(newUser);
            });
    }
}
07
Hardening & Best Practices

Security
Imperatives

OIDC is only as secure as its implementation. These are the non-negotiables for a production-grade system.

✅ Always validate the ID Token
Never skip signature, iss, aud, exp, or nonce validation. Spring Security does this automatically — never bypass it with custom logic. An unvalidated ID Token is worthless as proof of identity.
❌ Never use email as a primary key
Emails change and can be re-assigned. Use the sub claim — it's the stable, unique identifier for a user within an issuer. Store sub as your external identity FK.
✅ Check email_verified before trusting email
An unverified email could be anyone who typed it in. Always check email_verified: true before using email for password resets, notifications, or identity matching.
❌ Never put secrets in ID Token claims
The ID Token payload is base64url encoded, NOT encrypted. Anyone who intercepts it can decode the claims. Use JWE (JSON Web Encryption) if you need encrypted payloads.
✅ Use nonce for replay prevention
Always generate and send a cryptographically random nonce in your auth request. Validate it in the returned ID Token. Prevents token replay across different authorization requests.
✅ Implement step-up authentication
Check the auth_time claim before sensitive operations (deleting account, changing payment). If the user authenticated too long ago, redirect them to re-authenticate at the IdP.
🔐 Use RS256, not HS256 in production
RS256 uses asymmetric keys — the IdP signs with a private key, you verify with its public key (JWKS). HS256 uses a shared secret, meaning your app could forge tokens. Always prefer RS256/ES256.
🔑 Implement OIDC logout properly
Just clearing local session is not enough in SSO environments. Use RP-Initiated Logout to tell the IdP the user logged out, and implement Back-Channel Logout to handle IdP-initiated logout events.
❌ Don't trust claims from Access Tokens for identity
The Access Token is meant for API authorization, not identity. Its format and claims aren't standardized across providers. Always use the ID Token or /userinfo endpoint for user identity.
⚙️ Register exact redirect URIs
Configure precise redirect URIs at the IdP — no wildcards. Open redirect vulnerabilities allow attackers to steal authorization codes by registering a similar-looking URI. Be exact.
